# Contributing

感谢你对 `nonfiction-director-cut` 的兴趣。

这是一个用于非虚构内容中文导演剪辑版改写的模块化 Skill 项目。它不是普通提示词集合，而是由轻量入口、核心规则模块、模板、校验清单和迁移追溯文件组成的结构化系统。

为了保持项目稳定，请在提交修改前阅读本指南。

## 基本原则

本项目遵循以下维护原则：

1. `SKILL.md` 保持轻量，只负责触发、总控、优先级和模块路由。
2. 详细规则应放入对应的 `specs/` 文件。
3. 可复用输出结构应放入 `templates/`。
4. 校验、维护和错误记录应放入 `validation/`。
5. 迁移记录、历史追溯和方法论映射应放入 `migration/`。
6. 不要因为一次偶发输出问题直接修改核心规则。
7. 规则修改应尽量由真实问题、明确需求或维护证据驱动。

## 项目结构

```text
nonfiction-director-cut/
├── SKILL.md
├── specs/
├── templates/
├── validation/
└── migration/
```

### `SKILL.md`

根入口文件。

只应包含：

- Skill 触发说明
- 使用边界
- 核心行为
- 最小执行流程
- 优先级规则
- 模块加载说明

不要把详细写作规则、案例规则、输出模板或迁移过程说明加入 `SKILL.md`。

### `specs/`

核心规则层。

用于维护：

- 来源与模式判断
- 记忆与 RAG 规则
- 主问题提取与正文写作规则
- 案例策略
- 输出策略

如果修改会影响 Skill 的实际行为，通常应优先检查对应的 `specs/` 文件。

### `templates/`

模板层。

用于维护：

- 触发响应模板
- 正文骨架模板
- 可复用输出结构

模板不应定义新的核心规则。如果一个内容会改变判断逻辑，它不应只放在 `templates/` 中。

### `validation/`

校验与维护层。

用于维护：

- 主验证清单
- 蓝皮书参考检查清单
- 真实运行错误记录

`validation/mistakes-log.md` 只记录证据，不直接定义新规则。

### `migration/`

迁移追溯层。

用于维护：

- 原 monolith 迁移基线
- 蓝皮书参考映射
- 历史阶段记录

不要把新的执行规则写入 `migration/`。

## 可以提交的修改

欢迎以下类型的贡献：

- 修正错别字、格式问题或 Markdown 渲染问题
- 改善 README、CHANGELOG 或贡献指南
- 补充更清楚的使用说明
- 修复文件之间的引用错误
- 补充验证清单中的检查项
- 基于真实问题改进规则表述
- 提交经过说明的模板优化
- 记录真实使用中的失败案例或边界问题

## 需要谨慎提交的修改

以下修改会影响系统行为，需要提供充分说明：

- 修改 `SKILL.md` 的触发边界
- 修改权威层级
- 修改来源判断规则
- 修改模式判断规则
- 修改案例等级或案例禁区
- 修改防幻觉规则
- 修改输出边界或版权说明
- 拆分、合并或新增核心模块
- 新增进入日常执行路径的文件
- 删除已有规则或改变规则优先级

提交这类修改时，请说明：

- 修改动机
- 影响范围
- 是否来自真实使用问题
- 是否对应 `validation/mistakes-log.md` 中的记录
- 是否需要更新 `validation/validation-checklist.md`
- 是否需要更新 `validation/blue-book-checklist.md`

## 不建议提交的修改

以下修改通常不会被接受：

- 把详细规则重新塞回 `SKILL.md`
- 把模板当成规则源
- 把迁移记录当成执行规则
- 添加未经验证的现实案例
- 添加可能造成事实误导的示例
- 将本土化理解为机械替换外国案例
- 为了表达更吸引人而牺牲原素材忠实度
- 未经说明地改变输出风格、优先级或模块边界
- 添加大段受版权保护的书籍或文章原文
- 提交包含私人材料、用户材料、API key、账号信息或本地路径的文件

## 修改前检查

提交修改前，请检查：

```text
1. 这次修改是否改变 Skill 行为？
2. 是否改变规则优先级？
3. 是否改变模块边界？
4. 是否把详细规则加入了 SKILL.md？
5. 是否重复了已有规则？
6. 是否应该先记录到 validation/mistakes-log.md？
7. 是否需要更新 validation/validation-checklist.md？
8. 是否需要更新 validation/blue-book-checklist.md？
9. 是否包含未授权原文、用户材料或敏感信息？
10. 是否影响 README、CHANGELOG 或使用说明？
```

## 推荐提交流程

建议按以下流程提交修改：

1. 明确修改类型：文档、规则、模板、校验、维护或迁移记录。
2. 找到正确文件，不要把所有内容都放入 `SKILL.md`。
3. 如果是规则修改，先确认是否有真实问题或明确需求。
4. 如果是修复真实错误，优先在 `validation/mistakes-log.md` 中记录。
5. 修改后运行相关 checklist。
6. 在 PR 说明中写清楚修改动机和影响范围。

## PR 说明建议格式

提交 PR 时，建议使用以下格式：

```markdown
## 修改类型

- [ ] Documentation
- [ ] Maintenance
- [ ] Template
- [ ] Validation
- [ ] Rule change
- [ ] Module structure change

## 修改内容

简要说明改了什么。

## 修改原因

说明为什么需要这次修改。

## 影响范围

说明影响哪些文件、规则或使用场景。

## 是否改变行为

- [ ] 不改变行为
- [ ] 可能改变行为
- [ ] 明确改变行为

## 检查项

- [ ] 未把详细规则加入 `SKILL.md`
- [ ] 未引入重复规则
- [ ] 未改变未声明的优先级
- [ ] 未加入未授权原文或私人材料
- [ ] 已检查 `validation/validation-checklist.md`
- [ ] 已检查 `validation/blue-book-checklist.md`
- [ ] 如涉及真实错误，已记录或引用 `validation/mistakes-log.md`
```

## 版本记录

如果修改影响公开使用方式、文件结构、规则行为或维护流程，请同步更新 `CHANGELOG.md`。

如果只是修正错别字、格式或轻微说明，可以不更新版本号，但建议在 PR 中说明。

## 版权与素材边界

本仓库只应包含 Skill 指令、规则、模板、校验清单和维护文档。

请不要提交：

- 未授权书籍全文
- 未授权文章全文
- 付费内容大段摘录
- 用户私人材料
- 聊天记录中的敏感信息
- API key、账号、Cookie、Token
- 本地路径或内部系统信息

使用本 Skill 处理具体书籍、文章或其他材料时，请遵守相应内容的版权要求。
