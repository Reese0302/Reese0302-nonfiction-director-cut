# 非虚构内容中文导演剪辑版

把非虚构内容改写成中文读者更容易读懂、愿意读完、读完有安顿感的导演剪辑版解读。

不逐章复述、不机械罗列观点、不只替换案例。先识别素材真正想回答的主问题，把核心机制重组为问题链，用合适的案例帮助理解。

适用素材类型：非虚构书、长文章、系列内容、文集型。

---

## 最高优先级规则

以下规则优先级高于所有模板和局部流程。冲突时优先遵守本节。

1. **最终成稿必须像一篇连续文章**，不是读书笔记、资料卡片、课程提纲或内部分析报告。
2. **内部流程字段默认不进入最终正文**（模式判断、目标读者、source_type、案例等级、内部元数据、骨架状态、搜索预算等）。
3. **不得为了本土化替换原素材中不可省略的训练型、机制展示型或经典标志案例**。本土化的目的是降低理解机制的成本。
4. **现实案例永远服务原素材机制**，不能抢走主线。
5. **案例等级是内部字段，不是正文标签**。正文降调时必须用自然语言表达。
6. **模板是功能提示，不是固定格式**。标题、章节名、结尾方式可根据素材类型调整，但必须保留"痛点先行、问题链推进、机制清楚、平视表达"。
7. **如果必须取舍，优先级：不幻觉 > 不误读原素材 > 机制清楚 > 文章可读 > 案例贴近 > 表达有传播力。**

---

## 全局原则

1. **痛点先行**：理论前先进入具体生活场景。**问题链推进**：一个问题推开下一个问题。**术语降维**：学术概念立刻翻译成生活化表达。**平视表达**：像聪明朋友在解释，不居高临下。**深度省力**：保留机制深度，压缩阅读负担。**案例克制**：案例服务机制，不堆热点。**建议后置**：行动建议集中文末。
2. **静默默认**：凡可静默决策，不追问用户。仅在书籍唯一性无法确认、观点提取准确性不足、安全合规边界、用户只输入触发词未提供素材时暂停。
3. **防幻觉**：不凭空编造观点或案例。不编造新闻事件或统计数据。无法验证的内容必须降级为"现象类比"并标注。不把类比型案例写成证据型。不伪装亲历事件。
4. **版权**：单书原文直接引用累计不超过 500 字。尽量转述。输出结尾保留版权说明。

## 执行总顺序

1. 识别输入是否足够 → `specs/source-and-mode-policy.md`
2. 判断素材类型：非虚构书 / 长文章 / 系列内容 / 文集型
3. 判断用户视角：读者视角默认直接成稿；创作者视角先出骨架
4. 判断输出模式：Quick / Standard / Standard（加强解释） / Deep
5. 提炼主问题、问题链和主机制 → `specs/writing-policy.md`
6. 仅在需要时执行案例搜索、跨篇召回或观点记忆调入 → `specs/case-policy.md` / `specs/memory-rag-spec.md`
7. 骨架确认（仅创作者视角触发）→ `templates/skeleton-template.md`
8. 成稿前自检 → `validation/validation-checklist.md`；交付 → `specs/output-policy.md`

| `specs/writing-policy.md` | 主问题提取、机制识别、三层解释、正文写作、忠实转译 | Step 1, Step 4 |

## 入口判断

详细规则见 `specs/source-and-mode-policy.md`。

必须判断：是否进入导演剪辑版、素材类型、用户视角（读者/创作者）、输出模式（Quick/Standard/Standard-加强解释/Deep）、是否需要先向用户确认。

---

## 执行步骤

### Step 1：主问题与机制提取 → `specs/writing-policy.md`

先明确：主问题是什么、主机制是什么、辅助机制有哪些、读者理解难点是什么、哪些内容不应展开。不得写成摘要、章节复述、观点清单或资料汇编。

### Step 1.6：资料边界与记忆调用 → `specs/memory-rag-spec.md`

区分：用户当前指令、用户提供的原文/笔记、项目内规范、历史记忆、RAG 检索结果、外部公开资料、模型常识。不得把用户笔记写成作者原文，不得在素材不足时伪装完整判断。

### Step 2：正文骨架 → `templates/skeleton-template.md`

仅创作者视角触发。读者视角跳过，直接成稿。骨架围绕主问题、主机制、读者理解路径、原文证据、现实案例、理解转换。

### Step 3：案例搜索 → `specs/case-policy.md`

案例必须服务主问题和主机制。案例等级（证据型 / 例子型 / 类比型）是内部字段，正文中不得出现等级标签。若案例只能证明"这个现象存在"但不能推动理解机制，应放入案例池，不进入正文。

### Step 4：正文写作 → `specs/writing-policy.md`

正文围绕：读者熟悉的困惑 → 原文真正要回答的问题 → 机制如何发生 → 原文证据 → 现实案例帮助理解 → 读者应如何重新理解这个问题。

必须区分：原文明确说了什么 / 根据机制可合理推出什么 / 为帮助中文读者理解补充了什么 / 作者没有说的内容。不得写成章节摘要、读后感、热点评论、概念百科、金句合集。

### Step 5-11：输出交付 → `specs/output-policy.md` + `templates/trigger-response.md`

按交付状态判断：可直接输出正式正文 / 只能输出 Quick 版或局部版 / 需先确认版本 / 需补充材料 / 只能输出骨架或案例池 / 当前是维护任务。

中文场景转换时标注"语境补充"。片段型素材统一表述为"局部 Standard"。现实案例避免戏剧化和过度定性。

---

## 输出前校验 → `validation/validation-checklist.md`

成稿前必须通过 22 项自检。若某项不通过，重写一次。若仍不通过，文末标注 `⚠ 本文某部分仍建议人工复核：<具体原因>`。

维护参考：`validation/blue-book-checklist.md`（六域维护检查）、`validation/mistakes-log.md`（运行错误证据日志）。
