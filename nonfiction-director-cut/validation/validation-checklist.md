# Validation Checklist

来源：`SKILL.md` → `Step 9 输出前自检` + `验证方式`

迁移状态：已从 monolith baseline 迁移。

## 用途

本文件用于最终成稿前的内部质量检查，以及 Quick / Standard / Deep 三种模式的回归验证。

生成最终文章前，必须进行内部自检，不输出自检过程。

## 输出前自检

| 检查项 | 通过标准 |
|---|---|
| 主问题 | 是否是普通读者真实会问的问题 |
| 问题链 | 是否由 4-6 个问题自然推进，而不是观点并列 |
| 主机制 | 是否有 2-3 个详写主机制 |
| 辅助机制 | 是否有 2-3 个略写辅助机制，且不喧宾夺主 |
| 痛点先行 | 开头是否先进入生活困惑，而不是先讲理论 |
| 术语降维 | 每个关键概念是否有大白话解释 |
| 案例分级 | 每个现实案例是否完成内部等级判断；最终正文是否没有机械展示"证据型 / 例子型 / 类比型"标签 |
| 案例安全 | 是否避开隐私、高争议、个体苦难消费型案例 |
| 案例克制 | 案例是否服务机制，而不是堆热点 |
| 案例降调 | 类比型或弱相关案例是否用自然语言降调，避免被误写成证据 |
| 去模块化 | 最终正文是否没有出现「核心案例 / 原书案例 / 中国现场 / 现实案例」等硬模块标签 |
| 原书镜头融合 | 原书案例是否自然嵌入机制解释或问题推进中，而不是单独作为资料块出现 |
| 证据剪裁 | 每个主机制是否遵守对应模式的证据数量限制 |
| 追问式推进 | 每个主机制章节末尾是否有自然的追问式推进句 |
| 人格化判断 | 每个主机制章节是否至少有一句能命名感受、推翻旧解释或压缩机制的人格化判断句 |
| 成稿连续性 | 文章是否像连续成稿，而不是资料卡片、读书笔记或课程提纲 |
| 语气 | 是否平视、共情、不说教 |
| 篇幅 | 是否符合所选模式长度 |
| 行动建议 | 是否集中到文末，并且具体可做 |
| 版权 | 原文直接引用累计是否不超过 500 字 |
| 参考 | 是否保留原书信息和案例来源 |
| 结尾 | 是否回到主问题，并提供费曼式比喻 |
| 来源标注 | 若仅凭公开资料处理，是否已在文末标注"信息来源为公开资料，可能与原素材存在出入" |
| 加强解释 | 若采用 Standard（加强解释）写法：是否有至少一处把抽象判断翻译成了普通人能用的具体解释，且未写成价值口号 |

若某项不通过，重写相关部分一次。若仍不通过，在文末标注：

```markdown
⚠ 本文某部分仍建议人工复核：<具体原因>
```

## 模式验证方式

建议用同一本书分别测试 Quick、Standard、Deep 三种模式。

### Quick 验证

检查：

1. 是否没有直接长文展开。
2. 是否提炼出主问题。
3. 是否给出问题链骨架。
4. 是否只示范一个主机制。
5. 是否控制在 1500-2500 字。

### Standard 验证

检查：

1. 是否控制在 3500-5000 字。
2. 是否不是观点并列。
3. 是否有 2-3 个主机制详写。
4. 是否有 2-3 个辅助机制略写。
5. 是否有中文语境案例。
6. 每个现实案例是否已完成内部等级判断（证据型/例子型/类比型），且正文中未出现等级标签。
7. 行动建议是否集中在文末。

### Deep 验证

检查：

1. 是否展开更充分但不堆砌。
2. 是否有机制边界、误读风险和反直觉点。
3. 是否适合改成播客稿、课程稿或深度长文。
4. 是否控制在 6000-8000 字。
5. 是否仍然保持痛点先行和问题链推进。

### 风格验证

检查读完后是否产生以下感受：

1. 不是"我又被教育了"，而是"我被解释了"。
2. 不是"知道很多概念"，而是"一个困惑被重新组织了"。
3. 不是"案例很热闹"，而是"案例帮我理解了机制"。
4. 不是"读书笔记"，而是一篇重新剪过的中文解读。
