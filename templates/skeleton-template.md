# Skeleton Template

来源：`SKILL.md` → `Step 1.4 骨架输出模板`

迁移状态：已从 monolith baseline 迁移。

## 用途

当用户处于创作者视角，且任务需要先确认结构时，使用本模板输出「中文导演剪辑版骨架」。

本模板用于对齐以下内容：

1. 模式判断；
2. 主问题；
3. 问题链；
4. 主机制；
5. 辅助机制；
6. 暂不展开内容；
7. 案例方向。

## 触发条件

骨架确认环节由**用户视角**触发，不由模式触发：

- 创作者视角：Standard 和 Deep 必须先输出骨架并等待用户确认。
- 读者视角：跳过骨架确认，直接进入正文。

例外：当推荐模式与用户指定模式严重不匹配、且会显著影响成稿质量时，无论何种视角都允许用一句话温和提醒，但不强制等待确认。

如果用户未指定模式，骨架前必须先输出「模式诊断」。模式诊断和骨架合并在同一次回复中输出，不要强制多一轮确认。

## 骨架输出模板

```markdown
# 《书名》中文导演剪辑版骨架

版本判断：<书名、作者、版本、来源判断>

目标读者：<默认或用户指定>

## 模式诊断

推荐模式：<Quick / Standard / Deep / Standard（加强解释）>

判断依据：

1. 用户意图：<用户是想快速判断、自己学懂，还是用于发布 / 课程 / 深度内容生产>
2. 内容复杂度：<说明高分维度，例如机制依赖、方法密度、章节重组、误读风险、场景迁移等>
3. 解释成本：<说明为什么 Standard 足够，或为什么必须 Deep>
4. 不采用另一模式的原因：<如果不用 Deep，损失什么；如果不用 Standard，文章会不会过重>

你也可以改选：

- Quick：快速判断这本书讲什么、值不值得继续读；
- Standard：真正学到这本书最有用的核心；
- Deep：把这本书重构成一套完整理解系统。

如果你不修改，我将按推荐模式继续。

## 主问题

<一句普通读者真实会问的问题>

## 问题链

1. <问题一：从读者痛点切入>
2. <问题二：指出常见解释为什么不够>
3. <问题三：引出原书核心机制>
4. <问题四：放回中文语境会发生什么>
5. <问题五：这个机制有什么代价或误用>
6. <问题六：读者可以怎样重新理解自己或行动>

## 主机制

① **机制名** — <一句话解释>
   - 因果链：<A 导致 B，B 又放大 C>
   - 原书依据：<原书观点或案例，1-2 句转述>
   - 拟匹配中文语境案例方向：<具体方向>

② **机制名** — ...

## 辅助机制

① **机制名** — <为什么只略写>
② **机制名** — <为什么只略写>

## 暂不展开的内容

- <被舍弃的章节、概念或案例>：<舍弃原因>

请确认这个模式、主问题、问题链和案例方向。你可以改选模式、修改主问题、调整问题顺序、指定案例，或让我直接继续。
```

## 特殊规则

1. 如果用户已经明确指定模式，则骨架中的「模式诊断」可以省略。
2. 如果用户未指定模式，则必须保留「模式诊断」。
3. 如果推荐为 Standard（加强解释），最终元数据中的 mode 仍记为 Standard，但正文写法必须提高解释密度、减少口号化表达、强化现实翻译。
4. 模式诊断不得写成笼统判断，例如「这本书比较复杂」。必须具体说明复杂在哪里、为什么这个模式最合适、如果使用较低或较高模式会损失什么。
